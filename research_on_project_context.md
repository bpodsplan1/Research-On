# Research On — 프로젝트 컨텍스트 정리

**대상 조직**: 이트너스 BPO사업본부
**시스템 목적**: 주간 뉴스레터 자동 생성/발송 + 사내 리서치 허브(수동 키워드 조사 도구)
**구성**: n8n 워크플로우 4개 + 웹 프론트엔드(Research On 리서치허브, 단일 HTML) + Supabase(인증/DB) + Google Sheets/Docs(데이터 저장·맥락 문서)

> 이 문서는 최초 작성 이후 진행된 세션에서 있었던 변경사항을 반영해 갱신되었습니다. 최신 변경사항은 각 섹션에 통합했고, 연대기(7번 섹션)에 새 항목을 이어붙였습니다.

## 1. 전체 시스템 구성

| 컴포넌트 | 역할 |
|---|---|
| **뉴스레터 1단계** (n8n) | 매주 회사·클라이언트 관련 뉴스 + 트렌드 뉴스레터를 수집·평가해서 Google Sheets에 후보로 저장 |
| **뉴스레터 2단계** (n8n) | 1단계가 쌓아둔 후보 중 최근 것을 골라 실제 발송용 뉴스레터 HTML을 생성 + 관리자 승인(HITL) + 예약 발송 |
| **뉴스크롤링 1단계** (n8n) | 리서치 허브에서 사용자가 직접 키워드 검색할 때 쓰는 백엔드 (~20건 수집) |
| **뉴스크롤링 2단계** (n8n) | 리서치 허브에서 사용자가 선택한 자료를 심층 분석해 AI 인사이트 리포트를 생성 |
| **Research On 리서치허브** (HTML/JS) | 위 4개 워크플로우를 호출하는 프론트엔드. Supabase 인증, 단일 페이지 앱(SPA) 구조, 모바일 반응형 지원 |

## 2. 뉴스레터 1단계 — 후보 수집 프로세스

세 개의 하위 파이프라인이 병렬로 돌아가며, 각자 Google Sheets(`newsletter_research`)에 후보를 저장한다.

### 3-A / 3-B 파이프라인 (CLIENT_WATCH / BUSINESS_WORK)
```
검색 Job 생성(키워드 목록 기반, 최근 7일 날짜범위)
  → Naver + Serper 검색
  → 결과 통합 + 중복 제거(클러스터링)
  → 룰 기반 1차 점수화(target 매칭, DS/DX 필터, 발행일 등)
  → 발행일 10일 초과 기사 하드 컷
  → 점수순 정렬 후 섹션당 상위 16개만 추려 AI에게 전달
  → AI가 실제 이슈인지 판단(계열사 제외, 원문제목 재작성 등) 후 최대 6개 선정
  → CL4 이상 보직자(센터장·그룹장·사업부장·임원)를 겨냥한 작성 수준 지침 적용
    (요약 2~3문장 + 구체적 수치·시점 포함, 시사점은 "~기회입니다" 식 상투어 금지하고
     BPO사업본부 관점의 검토·논의 포인트를 담도록 지시)
  → Sheets 저장
```

### 3-C 파이프라인 (PEOPLE_CULTURE)
```
Gmail에서 최근 7일 트렌드레터 수집(뉴닉·캐릿·풋풋레터·까탈로그, 발신자 4곳)
  → 관리성 메일(인증메일·구독확인 등) 제외
  → 본문 정제 + 광고/협찬 블록 제거(by. 퍼블리 등) + 6000자 단위로 청크 분할(자르지 않고 전부 전달)
  → AI가 트렌드 후보 추출(최대 8개), search_query는 반드시 한국어로 작성하도록 지시
    (영문 트렌드명(Maxxing, Touch grass 등)을 그대로 쓰면 한국 뉴스 검색에서 결과가 거의
     안 나오는 문제가 있었음 — 프롬프트 지시 + 코드 레벨 안전장치 이중 적용)
  → Naver + Serper 뉴스 검색으로 후보 검증(예전엔 Serper만 사용했음)
  → 검색 결과 관련성 필터링: 키워드에서 "트렌드/소비/문화" 등 범용 단어를 뺀 핵심 토큰이
    제목+요약에 전부 등장하는지 확인 (Naver/Serper가 단어 단위로 느슨하게 매칭해서
    무관한 기사가 섞여 들어오는 문제 방지)
  → 관련 뉴스를 못 찾은 후보만 추가로 SNS(인스타그램/틱톡/threads/X) 대상 Serper 일반검색 시도
  → 검증된 자료만으로 최종 분석 작성 — 구체적 사례(예: "맥싱"이면 "파이버맥싱" 같은 하위 유형)를
    반드시 포함하도록 지시
  → organization_view(조직 운영 시사점)를 채용·유지·조직문화·세대 소통·복리후생 관점에서
    최소 2문장으로 필수 작성하도록 강화 (예전엔 "짧게 한 줄 또는 근거 약하면 빈 문자열"이었음 —
    이 때문에 실제 발송본에서 People & Culture 섹션에 조직 시사점이 통째로 빠져있었음)
  → Serper 응답의 imageUrl(base64 이미지 통째) 등 불필요한 대용량 필드는 이후 단계로 넘기기 전 제거
  → Sheets 저장
```

## 3. 뉴스레터 2단계 — 최종 발송본 생성 및 관리자 승인(HITL)

```
[A] Sheets에서 최근 7일 후보 정규화(섹션별 slice)
  → 회사 맥락 문서(900자, "본부 맥락") 읽기
  → 4개 꼭지(CLIENT WATCH·RESEARCH ON INSIGHT·BUSINESS & WORK·PEOPLE & CULTURE) 편집 프롬프트 생성
    (CL4+ 보직자 대상 작성 수준 지침 동일 적용, PEOPLE & CULTURE도 이제 implication 필수)
  → AI가 최종 기사 선정 + 제목 재작성 + 시사점 작성 (섹션당 여러 건 유지 — 예전엔 "웹 화면에서
    가장 앞에 남은 1개만" 강제로 잘랐던 로직이 있었음, 지금은 관리자가 검토 화면에 남긴 개수 그대로 전부 반영)
  → AI 응답 JSON 정리(URL 검증 등 안전장치 포함, 내부 리서치 결과 원문 링크는
    raw_json.source_summaries[].url에서 가져오도록 필드 매핑 버그 수정)
  → 관리자 승인용 후보 초안 반환

[B] 관리자 승인 및 발송 예약 (Human-in-the-Loop)
  → 관리자가 "AI 초안 가져오기"를 누르면 곧바로 실제 발송 템플릿 모습 그대로 렌더링됨
    (예전에는 폼 입력창 목록 + 별도 "미리보기" 버튼으로 분리되어 있었음 → 지금은 하나의 화면으로 통합)
  → 각 기사에 ▲▼(순서 변경) · 편집 · 삭제 버튼이 붙어 있어 그 자리에서 바로 수정 가능,
    변경 즉시 미리보기에 반영됨(스크롤 위치도 유지)
  → 섹션 끝에는 "+ 항목 추가" 버튼으로 새 기사를 바로 추가 가능
  → People & Culture "종합 코멘트"도 같은 방식으로 인라인 편집
  → 원문 기사 링크를 누르면: 인스타그램/틱톡/X/Threads/Facebook/YouTube 같이 iframe 삽입을
    서버 차원에서 막는 도메인만 새 탭으로 열리고, 일반 뉴스 기사는 미리보기 화면 안에서 그대로
    열림(이탈 시 "돌아가기" 버튼이 나타나고, 누르면 이탈 직전 스크롤 위치로 복귀)
  → 뉴스레터 본문은 word-break:keep-all 적용으로 한글 단어가 줄 중간에 끊기지 않음
  → "발송하기" 클릭 → n8n Webhook(`newsletter/queue`)으로 최종 데이터 전송
  → n8n이 승인본을 검증 + HTML 렌더링 + `newsletters` 시트에 QUEUED 상태로 저장 + 예약 접수 응답

[C] 예약 발송 실행
  → 매주 월요일 08시 스케줄 트리거로 QUEUED 상태 뉴스레터를 Gmail로 실제 발송
  → (테스트용 경로 추가됨) 관리자 승인 웹훅 직후 1분 뒤 자동 발송되는 별도 분기도 함께 존재
    — 운영 시엔 이 분기만 제거/비활성화하면 원래의 "월요일 08시만 발송" 구조로 복귀
  → 발송 완료 후 상태를 SENT로 업데이트
```

## 4. 뉴스크롤링 1/2단계 — 리서치 허브 백엔드

```
[1단계] 사용자가 키워드 조합 검색
  → Naver+Serper 검색(~20건) → 회사 맥락(600자)으로 관련성 평가 → 프론트엔드에 반환

[2단계] 사용자가 결과에서 자료 선택 후 "인사이트 도출"
  → 선택 URL 본문 추출(Tavily) → 추출 성공 여부 실제로 확인(과거엔 항상 true였던 버그 수정)
  → 본부 맥락(808자) 읽기 → 심층 리포트 프롬프트
  → AI 리포트 생성(Executive Brief, 사실요약, 분석, 추천액션, 참고자료)
  → 참고자료는 AI 응답과 무관하게 원본 문서의 실제 제목·URL을 코드에서 항상 보장하도록 처리
```

*(이번 세션에서는 이 두 워크플로우 자체에 대한 변경은 없었음. 뉴스레터 1단계 파이프라인이 이 2단계의
"AI가 URL을 지어내지 않도록 실제 검색 결과와 대조" 안전장치 패턴을 그대로 참고해 적용했음.)*

## 5. 회사 맥락 문서 체계

두 개의 문서를 용도에 따라 다르게 연결한다. (변경 없음)

| 문서 | 성격 | 길이 | 연결된 곳 |
|---|---|---|---|
| **A. 회사 맥락** | 회사 전체 차원의 넓은 정체성/서비스 설명. "이게 우리랑 관련 있나" 판단용 | 2000자(뉴스레터 1단계) / 600자(뉴스크롤링 1단계) | 뉴스레터 1단계 `0-C`, 뉴스크롤링 1단계 `8-A` |
| **B. 본부 맥락** | 센터·팀 단위 세부 조직/담당 직무. "그래서 누가 뭘 해야 하나" 판단용 | 900자(뉴스레터 2단계) / 808자(뉴스크롤링 2단계) | 뉴스레터 2단계 `4번`, 뉴스크롤링 2단계 `4-A` |

**본부 맥락(B) 핵심 내용**: 사업 개요(BPO 도급+ESOP), 조직/담당 직무(GA센터·BW센터·POC센터·ESOP워크센터·ESOP에듀센터별 담당 업무), 고객사(삼성전자 DS부문 우선·DX 후순위, ESOP 입주사는 별개), 전략 방향(Pain Point 기반 신사업, AX 정렬), AI 판단 지시(어느 센터·직무와 연결되는지 판단하라).

## 6. Research On 리서치허브 (프론트엔드) 주요 구조

- **인증**: Supabase, `@etners.com` 도메인 제한(클라이언트 단, 서버 RLS 미확인 — 보안 검토 필요 항목으로 남아있음)
- **사이드바**: 로고/아바타 대신 "OOO님 환영합니다" 텍스트 + 톱니바퀴(계정 설정) 아이콘. 로그아웃 버튼은 사이드바 하단에 flex 레이아웃으로 항상 보이도록 고정(순수 flex-column 구조 — `position:fixed` 오버레이 방식은 오히려 문제를 일으켜 되돌림). 메뉴 구성: 리서치 허브(대시보드/새 리서치 만들기/리서치 결과/AI 인사이트 리포트/저장 자료함/검색 히스토리), 뉴스레터 허브(뉴스레터/구독 신청), 관리자 메뉴(계정 관리/뉴스레터 관리/뉴스레터 구독 관리 — 관리자 권한 보유자만 노출)
- **모바일 반응형**: 840px 이하에서 사이드바가 슬라이드인 드로어(+백드롭)로 전환되고, 테이블 위주 UI(리서치 결과·인사이트 히스토리 등)는 카드형으로 전환됨. 데스크톱용(`...Table`)과 모바일용 컨테이너가 분리되어 있고 렌더링 함수가 두 곳 모두 채움
- **PC 전용**: 사이드바 접기/펼치기(아이콘만 남기고, 호버 시 body-level 툴팁으로 메뉴명 표시)
- **대시보드**: 1040px 폭으로 통일된 컨테이너 안에 브랜드 로고/검색창/바로가기 6개(카드형, 확대됨)/최근 본 리서치·오늘의 추천 키워드 카드가 배치됨. 예전엔 위쪽(브랜드/검색)은 640px로 좁고 아래쪽 위젯 카드는 화면 전체 폭으로 늘어나는 불일치가 있었음 → 통일
- **새 리서치 만들기**: 키워드 Pool(1/2/3-Depth) 탐색 → 전방 확장 → 후방 확장, 드래그 정렬 가능, 그룹별(전방/핵심/후방) 삽입 위치 규칙 적용. 고급 옵션에서 제외 사이트/키워드 직접 추가 가능(기존 프리셋과 도메인 정규화 비교 후 중복이면 자동 스킵)
- **AI 인사이트 리포트**: 목록(리스트) + 상세 페이지 분리 구조, Supabase에 누적 저장
- **뉴스레터 허브(구독자용)**: 구독 신청/해지 팝업(이메일 수정 가능), 다른 구성원 대리 구독 등록, 히스토리(발송본 iframe 미리보기, 한글 줄바꿈 방지 적용)
- **뉴스레터 관리(관리자 전용)**: 3번 섹션에 상술한 실시간 편집 미리보기 방식으로 전면 개편됨
- **계정 관리(관리자 전용)**: 계정 목록/권한 변경/비밀번호 초기화(백엔드 미구현, 토스트만)/삭제, 새 계정 발급 모달
- **뉴스레터 구독 관리(관리자 전용)**: 구독자 목록/발송 상태/일시정지·해제/구독취소

21. **PDF 잘림 버그 + Word 파일이 안 열리는 문제 수정, 뉴스레터 신규 구독 웰컴 메일 추가** (사용자가 실제 기기에서 다운로드해보다가 발견)
    - **PDF 잘림 버그**: AI 인사이트 리포트처럼 세로로 긴 콘텐츠를 PDF로 내보내면 특정 지점(스크린샷 기준 "리스크" 섹션)에서 잘리고 페이지도 1장으로만 나오는 문제. 원인은 페이지 분할(`sliceH`) 계산식이 실제 페이지 높이(mm)와 무관한 엉뚱한 공식(`canvas.width/ratio * (pageH/pageW) * 2`)이었던 것 — 세로로 긴 콘텐츠에서는 이 값이 지나치게 커져 "전체 내용이 한 페이지에 다 들어간다"고 오판하고, 실제 페이지 경계를 넘는 부분은 그대로 잘려서 안 보이게 됨. `pxToMm = pageW/canvas.width`, `sliceH = pageH/pxToMm`로 페이지 실치수 기준으로 정확히 재계산하도록 수정(리서치 결과 PDF 내보내기도 동일한 버그가 있어 같이 수정).
    - **Word 파일이 안 열리는 문제**: 원인은 애초 설계 자체의 한계 — "HTML 문서를 `.doc` 확장자로 저장"하는 오래된 트릭은 데스크톱 MS Word는 열어주지만 모바일 Word 앱·구글독스·한컴오피스 등 다른 프로그램은 손상된 파일로 인식해 열지 못함. `docx`(dolanmiu/docx, CDN: `cdn.jsdelivr.net/npm/docx@8.5.0/build/index.umd.js`) 라이브러리로 교체해 실제 유효한 OOXML(.docx) 바이너리를 직접 생성하도록 리포트·리서치 결과 Word 내보내기를 모두 재작성함(Node.js에 동일 라이브러리를 설치해 실제 코드를 그대로 실행 → 생성된 .docx를 LibreOffice로 직접 열어서 정상 변환되는지까지 검증 후 반영). 더 이상 쓰이지 않는 `reportTabsWordHtml()`은 제거.
    - **뉴스레터 신규 구독 웰컴 메일**: '구독 신청' 페이지에서 본인 구독 또는 타인 대리 등록(기존 계정/외부 게스트 양쪽 다)이 성공하면 새 웹훅(`newsletter/welcome`)을 호출하도록 프론트엔드 3곳에 연결. 신규 워크플로우 `[Research On] 뉴스레터 - 신규 구독 웰컴 메일 및 최신호 즉시 발송`을 제작 — 트리거 한 번이 두 갈래로 동시에 나뉘는 fan-out 구조(뉴스레터 2단계의 "승인 후 1분 대기" 분기와 동일한 패턴)로, (A) 웰컴 메일을 바로 보내고 즉시 웹훅 응답을 반환, (B) 이와 별개로 5분을 기다렸다가 `newsletters` 시트에서 가장 최근 SENT(없으면 QUEUED) 호를 찾아 그 구독자에게 바로 발송. 웰컴 메일 문구는 참고용으로 첨부하겠다던 타 뉴스레터 웰컴 메일 파일이 실제로는 첨부되지 않아 직접 새로 작성함(매주 월요일 오전 8시 정기발송 안내 포함) — 추후 참고 파일을 받으면 톤/포맷을 맞춰 다시 다듬을 수 있음.
22. **계정 관리 &gt; 새 계정 발급 모달에 부서/직위 필드 추가** — 기존에는 아이디/이름/이메일/역할/초기 비밀번호만 입력 가능했음. `profiles` insert 코드 자체는 이미 `affiliation`/`position` 컬럼을 받고 있었지만 항상 빈 문자열로 고정돼 있었던 것을 확인 — 계정 설정 페이지에 있던 것과 동일한 부서(조직도)/직위 드롭다운을 그대로 재사용해 입력 UI만 새로 추가함.

24. **사이드바 로그아웃 버튼이 다시 안 보이는 문제** (12번 항목 이후 재발) — 사용자가 예전에 로그아웃 버튼이 살아있던 별도 HTML 파일(`index3_logout_flex_bottom.html`)을 첨부하며 비교를 요청. 헤드리스 브라우저(Playwright)로 관리자 메뉴까지 포함한 최악의 케이스를 실제 좌표까지 측정해 비교한 결과, 현재 작업본의 레이아웃 구조 자체는 문제가 없었음 — 진짜 원인은 `.sidebar`가 `height:100vh`만 쓰고 있던 것. 모바일 브라우저에서 `100vh`는 주소창/툴바 뒤에 가려진 영역까지 포함해 실제 화면보다 커질 수 있어, 사이드바 맨 아래 요소(로그아웃 버튼)가 화면 밖으로 밀려나는 전형적인 버그였음. `height:100vh; height:100dvh;`로 폴백을 추가해 해결. (참고로 비교차 첨부받은 파일 쪽은 버튼 id가 `logoutDisplayBtn`으로 바뀌어 있는데 클릭 이벤트는 여전히 `logoutBtn`을 찾고 있어, 그 파일 자체는 버튼이 눌러도 반응이 없는 상태였음 — 현재 작업본은 이런 id 불일치가 없음을 확인함.)
25. **리서치 결과 상세 목록 UX 개선 + AI 인사이트 리포트 PDF 섹션 잘림 방지**
    - 목록의 각 항목을 체크박스를 정확히 눌러야만 선택할 수 있던 것을, 항목 전체(체크박스/버튼 제외 영역) 클릭으로 토글되게 변경. 제목을 눌러 상세 팝업을 열던 방식은 없애고, "저장" 왼쪽에 "상세보기" 버튼을 별도로 추가.
    - PDF로 내보낼 때 "기회 요인", "리스크" 같은 소제목 단위 섹션이 페이지 경계 중간에서 잘리던 문제 — `reportTabsPrintHtml()`이 만드는 각 소제목 블록에 `print-block` 마커를 붙이고, PDF 페이지 분할 시 그 블록 좌표를 미리 재서 경계가 블록 중간을 지나가면 블록 시작점에서 페이지를 끊도록 페이지 분할 로직을 재작성. html2canvas/jsPDF를 로컬에 내려받아 실제 코드를 헤드리스 브라우저에서 그대로 실행한 뒤, 생성된 PDF의 실제 페이지 분할 좌표를 블록 좌표와 대조해 섹션이 안 잘림을 수치로 검증함.
26. **회원가입 폼에 부서(전체 조직도)/직위 필드 추가 + 부서 드롭다운 표기 정리** — 회원가입 화면의 "소속"이 4개짜리 약식 목록(BW센터 1팀 등)이었던 것을 계정 관리 쪽과 동일한 전체 조직도 드롭다운으로 교체하고, "직위" 필드를 신규 추가. 이 과정에서 기존 부서 드롭다운(계정 설정/새 계정 발급/회원가입 3곳 모두)에 쓰이던 `-`, `.`, `*` 기호가 실제 UI 표기용이 아니라 계층 구조를 설명하려고 편의상 쓴 것이었음이 확인돼, 전각 공백만으로 들여쓰기하는 방식으로 3곳 다 정리. `handleSignup()`도 직위 값을 `profiles`에 저장하도록 반영.
27. **부서 선택 후 상위 조직이 안 보이는 문제** — 드롭다운에서 "1팀"을 선택하면 닫힌 select 박스에도 "1팀"만 표시돼 어느 센터 소속인지 알 수 없던 문제. select의 `value`에는 원래도 전체 경로(`BPO사업본부 &gt; DS지원사업부 &gt; GA센터 &gt; 1팀`)가 들어있었으므로, 목록 자체(들여쓰기된 짧은 이름)는 그대로 두고 select 아래에 선택된 전체 경로를 보여주는 작은 미리보기 텍스트를 추가(계정 설정/새 계정 발급/회원가입 3곳 모두, 공용 함수 `updateDeptPathPreview()`).

## 9. 작업 방식 원칙 (합의된 규칙)

- **워크플로우(n8n) 파일은 사용자 승인 없이 새로 만들지 않는다** — 먼저 대화로 방안을 정한 뒤 반영.
- 버그 수정은 **프롬프트 지시보다 코드 레벨 안전장치를 우선**한다(반복적으로, 프롬프트만으로 "더 정확하게/더 많이/덜 뭉뚱그리게" 요청하는 방식은 신뢰도가 낮다는 게 이 프로젝트에서 여러 번 확인됨). 이번 세션의 People & Culture 검색어 한국어화, 관련성 필터, `runOnceForEachItem` 누락 버그 등도 전부 이 원칙에 따라 코드 레벨에서 해결함.
- 수정 전 반드시 실제 데이터(첨부된 .eml, n8n 실행 결과 JSON, 관리자 화면 스크린샷 등)로 원인을 확인한 뒤 처방한다 — 추측성 수정 지양.
- 수정 후에는 실제 데이터 또는 재현 테스트로 검증한 뒤 전달한다. (CSS/HTML/JS 문법 검증, Node.js로 실제 함수를 실행해 결과 데이터 직접 확인, 필요 시 실제 렌더링 엔진으로 스크린샷까지 찍어 확인하는 방식을 반복적으로 사용함)
